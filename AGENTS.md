# 项目协作说明

## 项目背景

- 本项目基于 Next.js 和 Supabase 开发。
- 后续开发应优先遵循当前项目已有的技术栈、目录结构和代码风格。
- 修改前先阅读相关代码，尽量复用已有组件、工具函数和约定，避免无必要地引入新的依赖或并行方案。

## 项目文档目录约定

相关项目文档统一放在 `/docs/` 目录下：

- `/docs/prd.md`：存放项目需求文档、功能规格、已确认的产品决策，以及待确认问题。
- `/docs/images/`：存放截图、界面参考图、流程图等图片资料。
- `/docs/api/`：存放 API 文档、接口约定、第三方服务对接说明。

## 需求沉淀规则

- 当讨论功能实现、产品需求、业务规则或交互细节时，如果某些需求已经明确，应及时整理并写入 `/docs/prd.md`。
- 写入 PRD 前，只能基于已经确认的信息，不要把未经确认的猜测记录为最终需求。
- 如果需求还不完整，可以先记录为“待确认问题”或“开放问题”，并在后续确认后更新。
- 更新 PRD 时，尽量保持结构清晰，并标明功能规格、已确认决策和待确认事项。

## Supabase 操作规则

- 执行 Supabase 相关任务时，尽可能优先使用 Supabase MCP 完成操作。
- Supabase MCP 的优先使用场景包括但不限于：
  - 数据库表结构查看。
  - SQL 查询。
  - RLS 策略检查和创建。
  - Storage Bucket 的查看和创建。
- 涉及写入、删除、迁移或权限调整时，应先确认操作范围和潜在影响，再执行变更。

## 安全和配置约定

- 不要把 Supabase service role key、数据库密码、访问令牌等敏感信息写入前端代码或提交到仓库。
- 前端只能使用以 `NEXT_PUBLIC_` 开头的公开环境变量。
- 涉及用户数据访问时，需要优先考虑 RLS 策略和最小权限原则。
- 本地敏感配置应放入不会被 Git 提交的环境变量文件，例如 `.env.local`。
